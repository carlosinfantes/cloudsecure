CloudSecure

Agentless, serverless AWS security assessment platform. AI-powered analysis with professional compliance reporting.

Serverless Agentless AI-Powered Multi-Framework Python 3.12 AWS CDK

Platform Overview

CloudSecure is an automated cloud security assessment platform that scans AWS accounts for misconfigurations, compliance gaps, and security risks — without deploying any agents into the target account. It leverages STS AssumeRole for secure, temporary cross-account access and AWS Bedrock Claude for AI-powered finding synthesis.

Infrastructure

AWS CDK (TypeScript) — 4 stacks deployed to a single region. All resources are KMS-encrypted, serverless, and auto-scaling.

Compute

14 Lambda functions (Python 3.12) orchestrated by Step Functions. Prowler runs as a Lambda Container Image from ECR.

AI Synthesis

Amazon Bedrock Claude 3.5 Sonnet correlates findings, generates executive summaries, and prioritizes remediation actions.

CLI

Python CLI (pip install cloudsecure) with SigV4-signed API calls. Commands: assess, status, report.

Deployment Architecture

CloudSecure deploys as 4 interdependent CDK stacks. The Storage stack is the foundation, followed by Lambda, Orchestration, and API stacks that build upon each other.

Assessment Execution Flow

Each assessment is orchestrated by an AWS Step Functions state machine. The flow validates cross-account access, discovers resources, runs 7 analyzers in parallel, aggregates findings, applies AI synthesis, and generates reports.

Figure 2 — Step Functions State Machine: Assessment Execution Flow

Cross-Account Scanning Architecture

CloudSecure uses STS AssumeRole with ExternalId to securely access customer AWS accounts. No credentials are ever stored — only temporary session tokens are used during the assessment. Customers deploy a CloudFormation template that creates the required IAM role with read-only permissions.

Figure 3 — Cross-Account Scanning: STS AssumeRole with ExternalId

Onboarding Process

  1. Customer deploys onboarding/cloudformation/cloudsecure-role.yaml in their AWS account
  2. CloudFormation creates CloudSecureAssessmentRole with ReadOnlyAccess + SecurityAudit
  3. Trust policy specifies the CloudSecure platform account and a unique ExternalId
  4. Customer provides the Role ARN and ExternalId to CloudSecure via CLI or API
  5. CloudSecure validates access by assuming the role and testing permissions

Information Flow

Data flows from the CLI through API Gateway, gets processed by Step Functions, stored in DynamoDB and S3, and is returned as presigned report URLs.

API Reference

All endpoints require IAM authentication (SigV4 signed requests). The API is deployed as a regional REST API Gateway with CloudWatch logging enabled.

Security & Compliance

Encryption

  • KMS customer-managed key with rotation
  • S3 server-side encryption (SSE-KMS)
  • DynamoDB encryption at rest
  • Presigned URLs use SigV4 signing

Access Control

  • API Gateway: IAM authorization
  • Lambda: least-privilege IAM roles
  • Cross-account: STS AssumeRole + ExternalId
  • No credentials stored anywhere

Data Retention

  • 90-day TTL on assessment data (GDPR)
  • S3 lifecycle: Standard → IA → Glacier → Expire
  • CloudWatch logs: 30-day retention
  • Point-in-time recovery on DynamoDB

Compliance Frameworks

  • Active CIS AWS 1.4 / 2.0
  • Mapped NIST 800-53
  • Mapped ISO 27001, SOC2, GDPR
  • Planned HIPAA, PCI-DSS
